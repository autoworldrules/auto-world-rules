##WHAT EACH ARGUMENT LOOKS LIKE AND WHAT IT MEANS
All four arguments are Python strings. They describe a directed graph, a collection of graph-transformation patterns, and the extra directed edges that become implied implied from the existing explicit edges and the additional  candidate edge.

#ARGUMENT 1: cand_fact (str)
This is a single directed edge string you are scoring. It is exactly one edge statement ending in a period.
Example cand_fact:
maternal_aunt_or_uncle_of(2,3).
This is a directed edge of type maternal_aunt_or_uncle_of from node 2 to node 3.

Other candidate edges could be similar directed typed edges such as:
father_of(1,3).
living_in(3,4).
school_mates_with(0,1).
colleague_of(2,0).
not_living_in(1,4).

#ARGUMENT 2: definite_rules_program (str)
This is a string containing many graph-transformation rules, one rule per line. These rules describe when the presence of certain edge patterns in the graph implies that another edge must also exist.

Example definite_rules_program (small excerpt):
maternal_uncle_of(X,Y) :- maternal_aunt_or_uncle_of(X,Y), is_male(X,X).
aunt_or_uncle_of(X,Y) :- maternal_aunt_or_uncle_of(X,Y).
nibling_of(Y,X) :- aunt_or_uncle_of(X,Y).
living_in_same_place(X,Y) :- living_in_same_place(X,Z), living_in_same_place(Z,Y), !=(Y,X).
maternal_grandmother_of(V,X) :- paternal_grandmother_of(U,X), grandmother_of(V,X), !=(U,V).

Intuition: whenever there is a way to map the variables in the right-side pattern onto concrete graph nodes so that all required edges and constraints hold, then the left-side edge is implied between the corresponding concrete nodes.
Starting from the explicit edges already in the graph (the edges in facts_program together with the candidate edge cand_fact), these graph-transformation rules are applied recursively, so newly implied edges can trigger further rules and generate additional implied edges; 
the resulting hidden entailed edges are exactly the edges listed in entailed_facts.

Variables vs concrete nodes:
Capitalized tokens like X, Y, Z, U, V are placeholders for graph nodes.
Lowercase names like parent_of, is_person, aunt_or_uncle_of are edge types.
Concrete nodes in actual graph edges are integers like 0,1,2,3.
A variable assignment means choosing concrete graph nodes for the variables, e.g. assigning X=2 and Y=3, and then checking whether the entire right-side edge pattern exists in the graph under that choice; if it does, then the left-side edge is implied for those same concrete nodes.

Node constraints you may see include:
!=(U,V) meaning node U and node V are distinct.

Example graph-theoretic reading of a rule:
living_in_same_place(X,Y) :- living_in_same_place(X,Z), living_in_same_place(Z,Y), !=(Y,X).

This means:
if there are three nodes X, Z, and Y such that there is an edge of type living_in_same_place from X to Z, and an edge of type living_in_same_place from Z to Y, and Y and X are distinct nodes, then there is an implied edge of type living_in_same_place from X to Y.

#ARGUMENT 3: entailed_facts (str)
This is a string containing directed edges that become implied after adding cand_fact to facts_program and then applying the graph-transformation rules, excluding edges that were already explicitly present in facts_program (and excluding the candidate edge itself).

#ARGUMENT 4: facts_program (str)

For example, is_person(0). can be viewed as a self-loop of type is_person at node 0, and living_in(3,4). is a directed edge of type living_in from node 3 to node 4.

##WHAT MAKES A CANDIDATE EDGE USEFUL
A candidate edge is useful if it tends to produce at least one implied edge that is hard for reasoning models to recover from the explicit graph alone. You are not simply maximizing the number of implied edges. You are trying to favor candidates that unlock non-obvious consequences, often reflected by unusual graph interactions.

Because you will be shown a series of previously-evolved priority functions (priority_v0, priority_v1) that are already sorted by performance, those examples are your best guide to what kinds of heuristics are working in the evolutionary progression. def priority_v1(cand_fact: str, definite_rules_program: str, entailed_facts: str, facts_program: str) -> float:
```python
    """Prefer candidates whose entailed edges connect nodes that were previously far apart."""
    G = build_simple_graph(facts_program)  # graph BEFORE the candidate or its consequences
    entailed = text2edges(entailed_facts)
    if not entailed:
        return 0.0
    score = 0.0
    for src, tgt, rel in entailed:
        if not has_path(G, src, tgt):
            score += 5.0          # completely new reachability -- most surprising
        else:
            pl = shortest_path_length(G, src, tgt)
            score += pl           # longer detour before = more structurally surprising
    return score
```
